## Блокирующие правила (полный текст)

1. **WP Gate:** ЛЮБОЕ задание → `memory/protocol-open.md` → ДО начала работы. Новый РП (нет в плане недели) → Ритуал согласования (роль/работа/РП/переход состояния/режим ТВС/класс верификации/метод/оценка/модель) → дождаться явного «да»/«делаем»/«открывай» — БЕЗ этого не регистрировать РП в 4 местах (REGISTRY/WeekPlan/context/Linear). Полный алгоритм → `memory/protocol-open.md`.
2. **Push:** «заливай» / «запуши» / «закрывай» → commit + push без доп. вопросов. Push ДО отчёта Закрытия. **При любом Close-протоколе (Quick/Day/Week):** `git status --short` по ВСЕМ репо сессии — незафиксированные изменения → commit + push ДО перехода к следующему шагу протокола.
3. **Close:** Триггер Закрытия → протокол Закрытия → выполнить.
4. **Pull-on-Touch:** `git pull --rebase` при первом обращении к репо за сессию, один раз на репо, ко всем git-репо в `~/IWE/*`. Dirty → stash; конфликт/fail → процедура в `memory/reference/agent-core.md` (default: stash + пометка potentially stale → продолжить).
5. **Чеклист-верификация (Haiku R23):** Quick Close и Day Close — sub-agent Haiku R23 (context isolation) проверяет формальное соответствие чеклисту, не качество результата. Исключения: сессия ≤15 мин или без изменений файлов.
6. **Hooks/Scripts Bypass Gate (БЛОКИРУЮЩЕЕ, S-33):** Без **явного** разрешения пользователя НЕ менять скрипты шаблона (`.claude/hooks/`, `.claude/scripts/`, `.iwe-runtime/`, `FMT-exocortex-template/`) и НЕ обходить хуки. Хук блокирует → не обходить, зафиксировать bug-файл, сообщить пилоту, ждать инструкций. Полный алгоритм и исключения → `.claude/rules-lazy/hooks-bypass-gate.md`.
7. **Автономность (поведенческое):** НЕ спрашивать подтверждения («добавить?», «продолжить?», «хотите...?») — задание → выполни → отчитайся, без вопроса в конце. Очевидные следствия — делать сразу; проверяемый факт — проверять, не спрашивать. **Исключения:** необратимое разрушительное действие; WP Gate Ритуал (создание/закрытие/изменение РП); choice-question (выбор альтернатив, не yes/no).
8. **Напоминания (S-44):** «напомни через X», «поставь таймер» → `send_telegram_message` с `schedule_at` + одновременно ScheduleWakeup как резерв. При срабатывании резерва → сначала Telegram (немедленно, без `schedule_at`), потом чат. Нет Telegram-инструмента → только ScheduleWakeup + сообщить причину (зачем: Telegram доставляет на любое устройство, чат IDE — нет).
9. **Финиш > отлог (S-46):** обнаружена дополнительная задача в текущей сессии (косяк, недоделка, следующий шаг от субагента) → **дефолт = делаю сейчас**, НЕ «отдельный РП / технический долг» первым вариантом. Choice question «сейчас или потом?» = анти-паттерн (скрытое предложение отложить → inventory, не throughput). Исключения для отложения: бюджет ×2-×3, требуется доменное решение/данные, требуется ArchGate, контекст полностью переключился (другая часть системы). **WP Gate приоритет:** задача >15 мин + новый артефакт → п.1 действует (AR.001), п.9 не отменяет.

## Протокол Работы — полные абзацы hot-выжимки

**Capture-to-Pack** — на каждом рубеже: есть ли знание для записи? Анонсировать: *«Capture: [что] → [куда]»*. Маршрутизация: правило (1-3 строки) → CLAUDE.md, доменное → Pack, реализационное → DS docs/, урок → memory/. Capture-to-Pack — shortcut внутри маршрутизации знаний, не замена Routing Gate. При создании нового артефакта Routing Gate (DP.KR.001 §5) проверяется первым. Полный протокол → `memory/protocol-work.md`.

**Self-correction:** расхождение → немедленно предложить фикс (файл, строка, что изменить). Применяется только внутри scope текущего хода: файлы/директории из agenda хода (проверяется `git diff HEAD`). За пределами scope — Drift Reporting (Agent Core SYNC-CORE): отчитаться пилоту, не фиксить.

## §8 Staging — полный текст промоции

Промоция (Week Close): STAGING.md есть `validated`? → убрать авторские константы (`{{PLACEHOLDER}}`) → перенести в FMT + commit → статус `promoted`. Отклонение (специфично для авторского окружения) → статус `rejected`, остаётся в §9 навсегда — не удалять из таблицы, это решение. Пошагово → скилл `/author-mode`.
