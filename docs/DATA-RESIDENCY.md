# Резидентность персональных данных в IWE

> Дата: 2026-07-11 · Статус: принцип шаблона (WP-475)

**Если вы пришли сюда из [DATA-POLICY.md](DATA-POLICY.md):** тот документ — про данные, которые **платформа** собирает о вас (профиль, прогресс, цифровой двойник). Этот — про данные, которые **вы сами** приносите в свой IWE извне (здоровье, календарь, рабочее время) и про то, куда им можно и нельзя уходить.

---

## 1. Принцип

Данные, которые вы внесли в свой контур IWE, не покидают его без вашего явного согласия. Чтение данных, которые собирает платформа, внутрь IWE — это нормально и не требует отдельного разрешения на каждый раз. Вопрос не в том, откуда IWE читает данные, а в том, куда исходящий поток может уйти. Сборка (вычисление, генерация) на основе ваших данных всегда происходит на стороне вашего IWE, а не на платформе.

**«Резидентность» здесь — не географическое расположение данных** (не юрисдикция серверов, как в индустриальном термине data residency), **а правило контроля**: данные, которые вы внесли в свой контур IWE, не покидают его без явного согласия. Термин заимствован из другой области и здесь переопределён.

**«Контур IWE»** — это не физическое место и не степень доверия к сервису, а вопрос субстрата: контур — там, где физически версионируется дерево вашего личного IWE (`memory/`, `extensions/`, `params.yaml`, ваша стратегия). Это локальная машина плюс ваш приватный git-хостинг (GitHub, GitLab, self-hosted — любой). Всё остальное — платформенный Neon, Railway, WakaTime и другие сервисы — не субстрат IWE, а внешние точки: откуда-то можно читать (входящий поток, всегда можно), куда-то можно по явному согласию отправлять (исходящий поток, требует opt-in).

Пример: цифровой двойник в Neon — не часть контура, даже если IWE регулярно его читает (входящий поток, можно). Ваш приватный git-репозиторий — часть контура, потому что это и есть место хранения дерева IWE, а не пункт назначения для данных.

> Оговорка о терминологии: [DATA-POLICY.md §2](DATA-POLICY.md#2-где-хранятся-данные) называет серверную часть платформы «IWE (серверная часть)» в одной таблице с вашими личными файлами. В этом документе слово «IWE» используется только в значении «ваш личный экземпляр» — платформа отдельно.

## 2. Трёхслойная модель

Решение пилота, 10.07.2026:

| Слой | Что даёт | Чего не делает |
|------|----------|----------------|
| **Платформа** | Собирает свои данные о вашей активности внутри платформы (курсы, диагностика, события) и отдаёт их вам в IWE (входящий поток) | Никогда не становится транзитом или хранилищем для ваших локальных данных |
| **Шаблон IWE** | Даёт механизм (декларация потребности в данных + согласие + скилл-инвентаризация) и готовые рецепты для популярных источников | Не навязывает конкретный способ выгрузки данных |
| **Пользователь** | Сам решает и сам выполняет выгрузку своих данных — по рецепту из шаблона, своим скриптом или вручную | — |

Выгрузка данных с устройств и облаков — всегда ваше собственное действие. Ни платформа, ни шаблон не переносят данные за вас.

## 3. Типы личных данных

Источник: пост «Всезнающий ИИ есть у всех, а полезный только у тех, кто его самостоятельно построил» (шесть типов информации, из них четыре — личные):

| Тип | Что это | Где обычно живёт |
|-----|---------|-------------------|
| **2.1** | Устойчивые факты о вас (профиль, цели) | Частично платформа (входящий поток в IWE), частично ваши локальные заметки |
| **2.2** | Оперативный поток событий (здоровье, рабочее время, активность) | Обычно стороннее устройство/сервис → ваш локальный IWE |
| **2.3** | Формализованное личное знание (ваш Pack) | Ваш git-хостинг |
| **2.4** | Неформализуемая личная практика (хроника решений, интуиция) | Ваш git-хостинг (сессии, memory/) |

## 4. Стороннее облако — два разных случая

Инфраструктура, которая не платформа и не субстрат вашего IWE (см. §1), — тоже внешняя точка и тоже требует согласия на исходящий поток. Но у неё два разных случая с разным риском:

| Подтип | Пример | Кто управляет инфраструктурой | Согласие |
|--------|--------|-------------------------------|----------|
| **PaaS под вашим управлением** | Приёмник данных здоровья на Railway | Вы (свой аккаунт, свой инстанс, можете удалить в любой момент) | Требуется — данные физически покидают устройство, даже если инфраструктура «своя» |
| **SaaS-агрегатор** | WakaTime | Сторонний сервис — вы не управляете обработкой | Требуется — тот же принцип, риск выше: нет контроля над тем, как сервис использует данные дальше |

Оба случая — исходящий поток и требуют явного согласия. Более тонкая градация строгости согласия (например, разный UX для двух подтипов) — вопрос фазы реализации, не этого документа.

## 5. Формат «рецепта источника данных»

Конкретные источники данных оформляются как расширения шаблона (`extensions/`). Общий паттерн и схема декларации потребности в данных — предмет отдельной фазы; черновой пример уже намечен в одной из сессий WP-475.

### 5.1 Обязательные правила декларации (решения пилота 16.07.2026, WP-476 Ф1)

- **Версия схемы (`schema_version`) обязательна в каждой декларации потребности.** Декларация без версии не проходит валидацию — гейт останавливает функцию с внятной ошибкой, а не подставляет версию по умолчанию. Так реестр конвейеров строится только поверх выверенных схем.
- **Pre-grant список — только с явным подтверждением пилота.** Единственный постоянный носитель install-time разрешений — файл `pre-grant.yaml` рядом со скиллом гейта (`.claude/skills/residency-gate/`). Каждая запись обязана нести `approved_by: pilot` и дату `approved_at`; запись без подтверждения — блокирующая ошибка загрузки, функция сама себя в список добавить не может. Исходящие потоки (`flow_direction: outbound`) в pre-grant не попадают никогда. Детерминированная проверка: `python3 residency-gate.py validate-pre-grant` (кандидат в еженедельный аудит-скан Week Close).

## 6. Инвентаризация ваших данных

Планируется скилл, который поможет спросить: какие у вас есть данные типов 2.1-2.4 и где они лежат — вместо того чтобы каждый раз вручную придумывать интеграцию под конкретный источник. Место — `.claude/skills/` шаблона, рядом с другими скиллами инвентаризации; конкретное имя и полная реализация — предмет отдельной фазы.
