# Правила работы по типам репозиториев


## Лестница принципов и соответствие репозиториям

| Уровень | Что означает | Где живёт в IWE |
|---------|-------------|-----------------|
| ZPF (нулевые принципы) | Универсальные истины (логика, онтология) | Base / ZP |
| FPF (первые принципы) | Корректность по академическим дисциплинам | Base / FPF |
| SPF-методология | Форма домена для группы компаний (шаблон) | Base / SPF |
| SPF-инстанс | Наполненный домен конкретной компании | Pack |
| TPF-контентный | Планы, курсы, код — артефакты под орг | DS-репозитории (контент) |
| TPF-процессный | Как работает агент: ОРЗ, гейты, хуки, скиллы | DS-strategy + .claude/ + memory/ |
| LPF-декларативный | Утверждения, теряющие смысл при замене носителя данной роли на другого носителя той же роли | преимущественно `Lifework/`: «Личный устав», «Моя миссия», «Семейный уклад»; отдельные фрагменты «Профессиональных принципов» |
| LPF-оперативный | Фрагменты фактической траектории проживания роли (решения, сессии, планы), не сводимые к универсальному протоколу агента или доменному знанию — активные и архивные | фрагменты в `current/`, `decisions/`, `sessions/`, `inbox/WP-*/` и архивных аналогах (`archive/wp-contexts/`, `archive/decision-log/`, `archive/sessions/`) |

> *LPF (практики роли конкретного носителя, термин ailev, июнь 2026) — cross-cutting классификация артефактов/фрагментов, не контейнеров и не целых WP; применяется независимо от того, активен контейнер или архивный. Тест: фрагмент теряет смысл при замене носителя данной роли на другого носителя той же роли? Да → LPF; нет → PACK (доменное, переиспользуемое для роли вообще) или TPF-контентный (продукт для внешнего потребителя). Один WP/файл может одновременно содержать LPF-оперативный след (траектория решений) и TPF-контентный/PACK-выход (итоговый артефакт) — например, сама сессия WP-448 Ф11: диалог-траектория — LPF-оперативный, итоговая строка таблицы — TPF-контентный. LPF-декларативный — преимущественно в `Lifework/`, но не весь: «Манифест созидателей», «Конституция/Устав экосистемы» исключены тестом (сохраняют смысл при замене подписанта — это PACK/TPF-контентный, физически остаётся в `Lifework/` без переноса; перенос вне scope Ф11). Решено в peer-session `2026-07-02-03-wp448-f11-lpf-arch`.*

> *DPF (Domain Principles Framework, термин ailev 2026) = переименование SPF-инстанса в терминологии ailev. В IWE: DPF и Pack — два названия одной сущности. Формально: DPF = уровень принципов под конкретную организацию (SPF-инстанс). Содержательно: Pack = репозиторий с заполненным доменом. При чтении текстов ailev «DPF» → в IWE читать как «Pack». Переименование в IWE не планируется — только сноски (WP-448 Ф8).*

## 3 типа репозиториев

| Тип | Подтип | Что содержит | Source-of-truth | Кто создаёт |
|-----|--------|-------------|-----------------|-------------|
| **Base** | Принципы | ZP, FPF, SPF | Да | Платформа |
| **Base** | Форматы | FMT-* (протоколы структуры) | Да (для формата) | Платформа |
| **Pack** | — | Доменные знания (вторые принципы) | Да | Пользователь |
| **DS** | instrument / governance / surface | Производные от Pack | Нет | Пользователь |

> Base = платформа выдаёт. Pack и DS = пользователь создаёт.
> Pack = вторые принципы. DS = третьи принципы. Подробно: `ZP/README.md`

## Base (платформа)

### Base/Принципы (ZP, FPF, SPF)

**Содержит:** Нулевые принципы (ZP), фреймворки корректности (FPF, SPF), карта иерархии.

**Можно:** Уточнять формулировки принципов, обновлять карту иерархии, спецификации формы/процесса.

**Нельзя:** Доменное содержание (→ Pack). Дидактику (→ DS/surface).

### Base/Форматы (FMT-*)

**Содержит:** Протоколы структуры репо (FMT-exocortex-template, FMT-s2r).

**Можно:** Уточнять спецификацию формата, добавлять примеры.

**Нельзя:** Предметное содержание (→ Pack). Код (→ DS/instrument).

> Пользователь может создать свой формат — это будет DS с пометкой `template: true` в REPO-TYPE.md.

## Pack (source-of-truth) = вторые принципы

**Содержит:** Формализованные знания домена, методы, work products, failure modes, различения, SoTA-аннотации.

**Можно:** Добавлять/изменять знание домена, методы, WP, FM. Обновлять SoTA.

**Нельзя:** Дидактику (шаги, уроки). Путать метод с инструментом. Копировать без формализации. Хранить общие концепции FPF/SPF. Хранить планы/сроки/роли (→ Governance).

**Lint:** Проверяй process-lint перед коммитом.

> Pack содержит ТОЛЬКО доменное знание. FPF/SPF → Base/Принципы. Планы → DS/governance.

### Тест: нужен ли Pack вообще

**Тест на замену носителя роли** (LPF-критерий, WP-448 Ф11): формулировка держится, если заменить конкретного практикующего на другого носителя той же роли? Нет, держится только для этого человека (его тело, история, вкус) → не Pack, это LPF-контент (личная практика роли) → DS (Lifework-декларация или оперативная запись). Да, истина держится для роли вообще → Pack-кандидат.

**Кого именно заменять — исполнителя, не обязательно автора.** Носитель в тесте — тот, кто ФАКТИЧЕСКИ исполняет практику:
- Исполнитель = сам человек (тело, руки, голос — плавание, игра на инструменте) → заменять на другого ЧЕЛОВЕКА той же роли. Личная поправка обычно не переживает такую замену → LPF/DS.
- Исполнитель = делегат (ИИ выполняет работу вместо человека — генерирует текст/код/изображение в его стиле) → заменять на другого ИСПОЛНИТЕЛЯ-делегата (другую модель, будущую сессию). Формализованный стиль обязан пережить эту замену — иначе он непортируем и его пришлось бы заново извлекать из головы автора при каждой генерации. Такой стиль обычно тест проходит → Pack.

**Какая роль-режим автора чаще рождает Pack-контент.** Причина 1 (развитие домена) редко срабатывает в режиме «Профессионал» (применяет метод) или «Ученик» (осваивает) — там нет нового переносимого знания. Она срабатывает в режиме «Исследователь» (ищет новое) или «Просветитель» (формализует для передачи) — см. каталог ролей `PACK-personal/README.md`. Статус (профессионал/любитель) сам по себе не решает — решает, в каком из пяти режимов человек действует в момент создания контента.

**Три повода завести Pack (любого из трёх достаточно, не сумма):**
1. **Развитие домена** — формализуется новое знание, остающееся истинным для роли вообще, не только для автора.
2. **Личный метод, доведённый до нормативного** — сработавшая поправка к общему методу, которую хотят, чтобы ИИ применял как правило. Проверяется тем же тестом: поправка держится только для конкретного человека (тело, контекст) → не проходит тест, остаётся LPF/DS, даже когда автор формулирует её как «моё правило».
3. **Координация** — несколько потребителей (люди/агенты/артефакты) должны опираться на одно определение, иначе дублирование расходится (OwnerIntegrity).

**Частый ложный сигнал:** «ИИ производит из этого готовый рабочий продукт (текст/картинка/код) → значит нужен Pack». Неверно — работа генератора не меняет тест на замену носителя. Личный стиль, скормленный в пайплайн генерации, остаётся LPF-контентом; меняется не выбор Pack/DS, а требуемая строгость DS-артефакта (свободные заметки для совета vs формальная спецификация-параметр для автоматической генерации).

## DS (все подтипы)

> DS НЕ source-of-truth. Знание берётся из Pack.

| Подтип | Можно | Нельзя |
|--------|-------|--------|
| `instrument` | Код, тесты, документация кода | Менять source-of-truth |
| `governance` | Планы, реестры, решения | Менять предметное знание |
| `surface` | Курсы, гайды, маршруты | Создавать знание (только потреблять) |

### Правило минимума в DS/instrument

> **Репо агентов/ботов = преимущественно код и текст-как-код.**
> Архитектурные решения, UX-модели, доменные описания → Pack цифровой платформы.
> В instrument-репо остаётся: код, конфиги, CLAUDE.md (привязка к Pack), README (как запустить).

### Именование DS/instrument для ИИ-систем

Конвенция: `DS-{slug}` (примеры: `DS-strategist-agent`, `DS-extractor-agent`).
Привязка к Pack через `source-of-truth: {ID}` в CLAUDE.md репо, не через имя.

### DS/governance: DS-strategy (личная база данных)

**Содержит:** Планы недели, заметки (inbox/), контексты РП, extraction reports, онтологию.

**СТРОГО НЕЛЬЗЯ:** Скрипты, код, промпты, launchd-конфиги, агенты. Это ТОЛЬКО база данных — входы и выходы агентов. Агенты живут в своих DS-*-agent/ репо или DS-synchronizer/.


**Содержит:** Governance (реестры, планы, ADR, оргструктура) + операционная работа (семинары, предложения, черновики).

**Staging for Pack:** Операционный контент — промежуточная версия знания. На Close: KE извлекает формализованное знание → Pack, артефакт → архив. Это механизм самоочистки.

**Правило:** Формализованное знание домена НЕ хранится здесь постоянно — только транзитом через KE.

**0.9.Inbox:** только материалы к обсуждению с командой (повестки, предложения, вопросы). НЕ хранить WP-context файлы — они в DS-strategy/inbox/. Подробно: DP.KR.001 §5.3.


## Repository-first режим

**Никогда не отвечай из общих знаний без опоры на репозиторий.**

1. Найди минимум 1-2 релевантных файла в репозитории
2. Собери Context Pack
3. Только потом формируй ответ

Если контекст не найден → скажи "в репозитории нет источника", предложи создать документ.


## Context Pack (формат)

```markdown
## Context Pack для [запрос]
### Источники:
1. `путь/к/файлу.md` — описание
### Ключевые выдержки:
> [цитата с источником]
### Не использовано:
- Документ X — не релевантен, т.к. ...
```
